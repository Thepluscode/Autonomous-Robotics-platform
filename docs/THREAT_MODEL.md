# Threat Model v0.1 — How an Attacker Would Break Us, and What Stops Them

> **Audience.** Same as `METHODOLOGY_v0.1.md` — reviewers from Verra, Gold Standard, Climate Action Reserve, regen.network, Pachama, Sylvera, in-house compliance. Where the methodology document says *what we claim*, this document says *what would invalidate the claim* and *what we've done about it*.
>
> **Status.** v0.1 — pre-review. Gaps deliberately marked **[OPEN]**. This is the document we want adversarial pushback on.

---

## 1. What "the chain" must defend

The platform's only first-party security claim is that an observation, once signed, was generated by the platform at the timestamp embedded in its body, with the payload exactly as signed. Everything downstream (auditor verification, /gaia-prime stats, attestation roots) depends on that single property.

So the threat model is **scoped to the signed-observation chain**. We are not modeling:

- Generic web-app threats (XSS, CSRF, SSRF) that don't reach the signed bytes
- Availability attacks (DDoS) — the chain is append-only and survives downtime
- Compromise of *downstream* methodologies built on top of the chain (out of scope)

We *are* modeling: anything that lets an attacker produce a signed observation that an auditor would accept but that did not happen, or remove/alter a real signed observation without detection.

---

## 2. Trust model

| We trust | We do NOT trust |
| --- | --- |
| The Ed25519 algorithm (RFC 8032) | The signing service is the only signer — see §4.1 |
| The host clock (Railway production, NTP-synced) | Drone-reported timestamps |
| The Element84 earth-search STAC API as a *cross-witness* (i.e., a source we don't control) | Element84 as the *primary* source of truth |
| Auditors verifying with their own copy of `cryptography` and the published JWK | Auditors verifying only via our `/api/observations/verify` endpoint |
| Our own published public key at `/.well-known/keys.json` over TLS | The TLS PKI alone — auditors should also fetch the key out-of-band on first review and pin it |

The asymmetry matters: the chain stays defensible even if our backend is compromised tomorrow, *as long as the signing key is not exfiltrated*. A reviewer should evaluate the threat model with that distinction in mind.

---

## 3. Asset inventory (what's worth attacking)

| Asset | Where it lives | Compromise impact |
| --- | --- | --- |
| **Ed25519 signing key** | Production env var `OBSERVATION_PRIVATE_KEY_B64`, or HKDF-derived from `JWT_SECRET` | Catastrophic — attacker forges arbitrary observations indistinguishable from real ones |
| **JWT_SECRET** | Production env var (Railway) | Catastrophic in HKDF-fallback mode (derives signing key); auth bypass + signing key |
| **MongoDB write access** | Internal-network reachable from backend pod | Selective tampering: deletion of observations, modification of unsigned fields (`metadata.redacted`) |
| **MongoDB read access** | Same | Privacy (poaching coordinates, PII) — addressed in `DATA_HANDLING.md` not here |
| **Admin auth cookie / JWT** | Browser of `admin@…` user | Trigger redact-flag on signed observations (signed bytes unchanged; visibility toggled) |
| **Backend host** | Railway service `backend-production-0e26` | Capability to call `record_signed_observation()` with arbitrary payload — same impact as signing-key compromise for the duration of the breach |
| **Drone fleet** (today: simulated) | Backend `simulator.py` tick loop | None today — simulated. **[OPEN]** Real drones are the obvious next attack surface; see §4.4 |

## 4. Threat vectors

Each vector lists: what the attacker can do, current mitigation, residual risk.

### 4.1 Signing key compromise (catastrophic)

**Attack.** An attacker obtains `OBSERVATION_PRIVATE_KEY_B64` (or `JWT_SECRET` when HKDF fallback is active) and signs arbitrary observations offline.

**Mitigation today.**
- Production uses `OBSERVATION_PRIVATE_KEY_B64` — a dedicated 32-byte Ed25519 key, *not* derived from `JWT_SECRET`. Compromising the auth secret no longer compromises the signing key.
- `JWT_SECRET` startup validator (`backend/server.py::_validate_jwt_secret`) refuses to boot on missing / blocklisted / <32-char values, so the HKDF fallback at least never runs on a weak secret.
- Keys live in Railway env vars, not in source. `.env` is git-ignored.

**Residual risk.**
- **No key rotation procedure documented.** A leaked key invalidates the chain. v0.2 candidate: publish two JWKs (current + previous) and a `rotate_at` timestamp so verifiers can validate observations against the correct key for their era.
- **Single signing operator.** See methodology §6.1.
- **[OPEN]** Hardware-backed signing (YubiHSM, cloud KMS) is not deployed. This is the single highest-leverage hardening we have not yet done.

### 4.2 Backend compromise (catastrophic but bounded)

**Attack.** Attacker gets RCE on the backend pod. They can call `record_signed_observation()` directly and produce *real* signatures, indistinguishable from organic activity.

**Mitigation today.**
- The Phase-A auth gate (`AUTH_GATE_PHASE_A=1`) reduces unauthenticated reachable surface to the `PUBLIC_ROUTES` allowlist. Lateral movement from a leaked viewer cookie is bounded by role.
- `JWT_SECRET_ALLOW_INSECURE` cannot quietly survive into a deploy — every boot with the escape hatch logs `CRITICAL`.
- The public-surface lock test (`tests/test_unit.py::test_public_surface_is_locked`) prevents accidental exposure of new internal routes.

**Residual risk.**
- A *real* backend compromise still lets the attacker sign arbitrary observations for the duration of the breach. The cross-witness in §4.5 is the answer; not a perfect one.
- **[OPEN]** No append-only audit log of *who triggered which signature*. v0.2: a side-channel log of `(caller, observation_id, source_type, host)` that an auditor can reconcile against the chain.

### 4.3 Insider / operator forging observations

**Attack.** The platform operator (us) signs observations that didn't happen. We have the key by design.

**Mitigation today.**
- **Satellite cross-witness** (`backend/satellite.py`). For zones with bounded geometry, Sentinel-2 thumbnails are hashed and signed independently of the operator-controlled drone stream. An auditor checking that the operator's drone telemetry agrees with the operator-controlled-but-independently-verifiable satellite witness raises the cost of fraud to "lie consistently across two independent infrastructures."
- The satellite witness pulls from a source the operator does not control (ESA / Element84). The auditor verification flow in `METHODOLOGY_v0.1.md` §2 lets a reviewer independently re-fetch the same Sentinel-2 scene and compare hashes.

**Residual risk.**
- **The operator chooses *which* zones to monitor.** Fraud-via-omission (don't sign anything for a zone where the truth is inconvenient) is not addressed by §4.3 alone. The aggregate-root publication cadence helps; an auditor noticing a zone's chain stops growing is the detection signal.
- The satellite witness loop is **default-OFF** in dev and could be silently disabled in production. v0.2 candidate: publish the satellite-loop's last-tick timestamp on `/api/public/provenance/stats` so a reviewer notices if it stops.
- **[OPEN]** A real multi-operator chain (methodology §6.1) is the structural fix. We have not built it.

### 4.4 Compromised drone / sensor (today: hypothetical; soon: real)

**Attack.** A drone is physically tampered with, its firmware replaced, or its uplink MITM'd. It reports fabricated telemetry that the backend signs in good faith.

**Mitigation today.**
- **None directly.** The drones are simulated; there is no real fleet to compromise. This is the methodology's §6.1 limitation made concrete.
- The signed-observation chain records *what the drone said*, not *that the drone was honest*. An auditor reading the chain must treat unwitnessed drone telemetry as no stronger than the drone itself.

**Residual risk.**
- **This is the largest known unmitigated vector.** Three structural fixes are candidates:
  1. **Drone-side signing.** Each drone holds its own Ed25519 key; the backend co-signs an already-signed message. Tamper detection requires breaking the drone's secure enclave.
  2. **Behavioral baseline / UEBA.** Profile normal patrol pattern per drone (velocity envelope, telemetry-rate distribution, sensor-reading ranges) and reject observations outside the envelope *before* signing. Roadmap idea inherited from CyberGuard research; explicitly deferred until auditor feedback says it's required.
  3. **Cross-witness for terrestrial observations.** Satellite covers the zone-scale claim. Per-drone telemetry has no current cross-witness. v0.2 candidate: pairwise drone witness (two drones in the same zone confirming each other's position).
- **[OPEN]** This vector is also the most likely to come up in an auditor's first 5 minutes. We expect to be told which of (1)–(3) is the table-stakes fix.

### 4.5 Cross-witness service compromise

**Attack.** Element84's earth-search STAC API is compromised; an attacker manipulates Sentinel-2 thumbnail bytes for a specific scene to match a fabricated narrative.

**Mitigation today.**
- The satellite witness is *additive*, not *substitutive*. An auditor verifying a claim re-fetches the canonical STAC item from `payload.stac_url` (Element84) AND can independently fetch the same Sentinel-2 L2A scene from ESA / Copernicus Data Space (where Element84 mirrors from). Two-step verification.
- Cloud-cover filter (`SATELLITE_MAX_CLOUD_COVER=60`) prevents adversarial cloud-cover gaming in the *witness* direction. **[OPEN]** It does not prevent witness-side cloud-cover *manipulation* (claiming a real high-cloud scene as low-cloud).

**Residual risk.**
- An attacker who compromises *both* the platform operator and Element84 simultaneously can forge a coherent cross-witness. Realistic adversaries (state actors targeting forest-carbon fraud) make this non-zero but vanishingly small.
- **[OPEN]** A second independent satellite source (Planet Labs, Maxar, or Sentinel-1 SAR for cloud-immune cross-witness) is the v0.2 hardening for high-stakes claims.

### 4.6 Replay attacks

**Attack.** An attacker captures a previously-signed observation and re-submits it as a current event.

**Mitigation today.**
- Every observation includes `observed_at` (ISO-8601 UTC) inside the signed body. Replaying it does not change the embedded timestamp.
- `GET /api/observations` returns observations indexed by their original `observed_at`, so a replayed copy appears at the original timestamp, not "now."

**Residual risk.**
- A replayed observation submitted to a *separate* methodology pipeline that ignores `observed_at` could be miscounted. This is a downstream-methodology concern, not a chain concern, but reviewers should flag it.

### 4.7 Tampering with unsigned metadata

**Attack.** The signed bytes are immutable, but the Mongo document also has *unsigned* metadata (`metadata.redacted`, indexing fields, search tags). An attacker with Mongo write access flips `metadata.redacted=true` to hide an observation, or modifies a search tag to break filtering.

**Mitigation today.**
- The redact endpoint (`POST /api/observations/{id}/redact`) is admin-gated. Direct Mongo writes from outside the API bypass this — that's a server-compromise vector, not a redact-endpoint vector.
- Aggregate roots (`GET /api/zones/{id}/attestation`) are computed over signed-body digests. A redacted observation **does not** drop out of the aggregate; the root still includes its digest. So tampering-via-redaction is detectable by re-computing the root from the public observation list.

**Residual risk.**
- A *deleted* (not redacted) observation drops out of both the listing and the aggregate. If an auditor doesn't snapshot the aggregate root at time T, they have no way to detect a later deletion. **[OPEN]** v0.2: publish aggregate roots on a fixed cadence to an external pinning service (IPFS CID, public Git tag, OpenTimestamps) so historical roots are auditable.

### 4.8 Aggregate root v1 weakness

**Attack.** The aggregate root is `SHA-256` of the sorted, newline-joined list of observation digests in the window. An auditor checking that a specific observation is "in" the root must enumerate the entire list.

**Mitigation today.**
- The list is public (`/api/observations`). The check is mechanical: re-fetch the list, sort, hash, compare.

**Residual risk.**
- No succinct inclusion proof. For zones with >10⁴ observations per window the cost of enumeration becomes nontrivial. Methodology §6.6 already calls this out; the threat-model implication is *availability of the verification path*, not chain integrity.

### 4.9 Public-endpoint abuse

**Attack.** Unauthenticated bulk-scrape of the chain, used to enumerate sensitive locations (poaching GPS, intervention sites).

**Mitigation today.**
- `ATTESTATION_MAX_HOURS = 168` cap on the attestation endpoint prevents single-call bulk export of >7 days.
- Redact flag (admin-gated) for observations whose disclosure would create safety risk — see `docs/DATA_HANDLING.md`.

**Residual risk.**
- **No rate limit on `/api/observations` or `/api/public/provenance/stats` today.** A patient scraper can paginate the full chain. **[OPEN]** v0.2: per-IP rate limit on public endpoints (e.g., 60 req/min anonymous, 600 req/min with API key).
- The redact mechanism is *operator-controlled* — see §4.3. An adversarial operator can over-redact; an auditor noticing high redact rates is the detection.

## 5. Out-of-scope (deliberately)

We are not modeling the following, and a reviewer pushing back on this is welcome:

- **Quantum-resistant signatures.** Ed25519 is broken by sufficiently large quantum computers. The 10-15 year horizon is well beyond the methodology's current versioning discipline. v3 problem.
- **Side-channel attacks on the signing host.** Power analysis / timing attacks against the Railway pod are below the realistic-adversary threshold for biodiversity-credit fraud.
- **Sybil attacks at the auditor level.** If an auditor signs off on a fraudulent claim, the chain still records what was signed; that's a governance problem, not a cryptographic one.
- **Insider attacks against the auditor.** Out of scope by definition.
